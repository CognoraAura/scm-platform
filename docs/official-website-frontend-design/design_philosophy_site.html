I Collaboration Is the Primary Abstraction
Most systems treat collaboration as a layer added on top of a storage system. You upload a file, the system stores it, and collaboration is grafted on — comments here, version history there, real-time cursors as a premium feature.
When collaboration is primary, the data model is shaped around concurrent access from the start. When it is added later, you get locks, optimistic concurrency errors surfaced to users, and "someone else is editing this" warnings — which are not UX failures. They are architectural failures that have leaked into the interface.
The inversion means: a single user uploading an asset in isolation is a degenerate case of the collaboration model, not the primary case that the collaboration model is bolted onto.
II Why Yjs CRDT — and Why Not OT
The choice of synchronization primitive is the most consequential decision in any collaborative system. Get it wrong and you spend years fighting your consistency model.
Decision record
rejected Operational Transformation. Requires a central server for canonical operation ordering. That server is a coordination bottleneck. Transformation functions for rich structured data are combinatorially complex and notoriously difficult to prove correct.
chosen Yjs CRDT. Data structures closed under a commutative, associative, idempotent merge function. Any two replicas merge independently and arrive at the same state, regardless of operation order. Convergence is a property of the data structure, not the implementation.
cost Yjs document state is binary and opaque. Inspecting state requires a deserializer. Server-side merge requires GraalVM Polyglot to execute the Yjs JavaScript merge function. Real operational overhead that teams consistently underestimate.
→
Offline resilience
A client accumulates edits while disconnected and merges on reconnect without coordination. The server does not need to know the client was offline.
→
Peer-to-peer capability
The server is a relay and persistence layer, not a merge authority. The architecture doesn't use P2P exchange — but the property constrains us toward better designs.
→
Mathematical correctness
You cannot write a bug that causes divergence without breaking CRDT invariants. Correctness is structural, not behavioral.
III Event-Driven Synchronization — Philosophy, Not Pattern
A system that communicates through events is a system where operations complete when they are recorded, where consumers discover events rather than being wired to producers, and where the event log is a first-class artifact that can be replayed, audited, and extended.
Event-driven architecture is often adopted as a pattern: "we should decouple these services." That framing is too shallow. We adopted it as a philosophy about what the system should be able to observe, replay, and evolve.
The event as boundary: Every domain event is a boundary between what the system knows and what it is doing about it. AssetIngested means the system knows this asset exists durably. What it does about that fact — tagging, deduplication, notifications — is downstream of that knowledge, not part of it.
Rejected alternative
rejected Synchronous pipeline. An AI API timeout at 15 seconds becomes the user's upload latency. A moderation service outage blocks all uploads. Adding a new enrichment step requires changing the upload handler and its error handling. Failure modes are unacceptable.
cost Asynchrony is harder to debug. Events can be lost. Consumers can fail. Ordering is not guaranteed. Managed through idempotent consumers, retryable stages, and correlation IDs that make causality traceable across async boundaries.
IV Cache-Aware Media Delivery — First Principles
A cache is not a performance optimization bolted onto a correct system. In a media delivery system, the cache architecture is the performance architecture. Getting it wrong means the correct behavior is also the slow behavior.
L1
Caffeine — JVM heap, ~1µs
~10k entries, TTL 5 min. The hot set — top assets by access frequency. For an active campaign, everything the team looked at in the last hour is here. Zero Redis RTT.
L2
Redis — shared across pods, ~1ms
~500k entries, TTL 30 min. Warm set. A cache warm from pod A is immediately available to pod B. Presigned URLs aligned to URL expiry minus safety margin.
⊘
Write-around strategy — not write-through
Batch uploads are read-rarely-after-write. Pre-warming the cache on upload evicts genuinely hot assets. Write-around accepts slightly higher latency on first access; the cache reflects actual access patterns, not upload patterns.
✕
Strong invalidation on mutation
On asset deletion or version update, both L1 and L2 are synchronously invalidated. Stale window is zero. We pay a synchronous Redis write on every mutation — correct for a read-dominated system.
V Storage Abstraction — The Future-Proofing Argument
Storage backends change. Not because engineers are indecisive — because the economics change, the compliance requirements change, the deployment targets change, and the vendors change their pricing models.
A system that has Tencent COS SDK calls distributed across its business logic has made a permanent architectural commitment to Tencent COS. The ObjectStoragePort interface is a one-time investment that makes future migrations cheap.
Interface contract
principle The interface encodes what the business logic needs, not how a specific backend provides it. presignedGet(key, ttl) is a business concept. Whether the URL is signed by COS, S3, or a local file server is the adapter's concern.
rejected Direct SDK integration. Faster to write initially. The problem is what happens over two years: the team discovers a new deployment requirement, and the system has COS assumptions scattered through twenty service methods.
cost Two additional hours of initial implementation time. Saves weeks of migration work later. The math is not close.
VI AI as Ingest Infrastructure
The conventional approach: a user selects assets, clicks "Generate Tags," waits for a spinner, and reviews the results. This treats AI as a user-invoked tool. It is better than nothing. It is not good enough.
When AI tagging is part of the ingest contract — an unconditional pipeline stage that runs on every asset — the system has a consistent enrichment baseline. Every asset is tagged. The search index has consistent coverage. Deduplication signals are available for every upload. The user sees better results without taking any additional action.
→
Unconditional — no opt-in
When tagging is a user action, most assets are never tagged. Manual actions are often skipped. The ingest pipeline removes the human decision from the loop for enrichment that should always happen.
→
Async — off the upload critical path
Qwen-VL inference for a high-resolution image can take seconds. Upload returns when storage is durable. Enrichment completes within seconds. The asset is usable immediately.
→
Taxonomy normalization at the pipeline boundary
The AI call is tenant-agnostic — the same model serves all tenants. The normalization stage maps raw semantic labels to the tenant's vocabulary. Separation of concerns at the pipeline boundary, not the API call.
VII Multi-Tenant Isolation — Defense in Depth
Multi-tenancy is not a feature. It is a security property. The consequences of getting it wrong are not performance degradation or a confusing UI — they are data breaches, compliance failures, and loss of customer trust that cannot be recovered.
Isolation layers
primary Hibernate filter adds WHERE tenant_id = ? to every query on tenant-scoped tables. Activated at session creation. Cannot be disabled from application code. The defense is in the infrastructure, not in the developer's attention.
storage Storage key prefix — {tenantId}/{year}/{month}/{assetId}/{variant}. A signed URL for tenant A cannot access tenant B's data. Also enables per-tenant cost attribution at the billing level without querying the database.
enterprise Schema-per-tenant via routing DataSource for regulated tenants. Application code is identical. Routing happens at the JDBC connection level. Available for compliance requirements that demand physical data separation.
The threat model is not a sophisticated attacker who has compromised the application server. It is an application bug — a missing filter clause, a misconfigured query, a shared cache key — that allows one tenant's request to return another tenant's data. These bugs are the most common multi-tenancy failure mode, and the hardest to catch in testing.
VIII What This System Does Not Solve
A system's non-goals are as important as its goals. A system that tries to solve everything solves nothing well.
✕ Video processing. Video is a different problem domain — transcoding pipelines, streaming protocols, adaptive bitrate delivery. Adding video would mean a media processing infrastructure that dwarfs the existing system in complexity.
✕ Global strong consistency. The consistency model is deliberately mixed. A system that requires global strong consistency across all operations cannot achieve the latency targets that make real-time collaboration feel real.
✕ Cross-region replication at the application layer. The storage abstraction supports a multi-region-capable backend. The application does not implement region-aware routing. Cross-region replication is a deployment decision, not an application feature.
✕ Public anonymous delivery. Presigned URLs provide authenticated, time-bounded access. Building a public CDN with anonymous access and edge caching is a different product with different security properties.
✕ Pixel-level collaborative drawing. Figma's problem. This system solves the asset management problem adjacent to it, not the canvas collaboration problem itself.
IX Scaling Philosophy
Scale on demand, not in advance. Every premature scaling decision adds complexity that must be maintained, debugged, and explained to new team members. Add complexity when the problem is demonstrated, not when it is anticipated.
scales horizontally
HTTP API tier
Stateless pods. JWT-carried identity. No shared process state. Adding a pod requires zero coordination.
harder problem
WebSocket / STOMP tier
Stateful connections. Redis pub/sub serves as cross-pod broadcast. Introduces Redis availability dependency on the collaboration path.
scales independently
AI pipeline workers
Stateless event consumers. Adding workers adds throughput. The event bus is the backpressure mechanism.
deferred
Event bus + PostgreSQL
In-process event bus by default. PostgreSQL not sharded. The DomainEventPublisher interface and a well-tuned PG instance are the seams. Cross them when the ceiling is reached.
X Operational Philosophy
A system that is correct but unoperatable is not correct in production. Operational concerns are not second-class — they are part of the design.
①
Observability is not optional
Structured logs, distributed traces, and business metrics are written alongside the features they observe, not added after the system works. A feature that cannot be observed is not complete. Every async operation carries a correlation ID that makes causality traceable across service boundaries.
②
Idempotency is a contract
Every pipeline stage is idempotent. Running a stage twice produces the same result as running it once. In a distributed system, at-least-once delivery is easier to achieve than exactly-once. We design for at-least-once by making consumers correct under repeated delivery.
③
Failure is normal — degrade gracefully
The system is designed to fail gracefully, not to prevent failure. AI API timeout → retry, fall back to untagged. Storage error → fail upload, do not emit domain event. Redis unavailable → fall through to PostgreSQL, degrade on WS broadcast. A system that has not specified its failure behavior has not finished its design.
― Closing
Every opinion in this document is falsifiable. We believe CRDT convergence is worth the operational cost of binary document state. We believe write-around cache strategy is correct for upload-heavy creative workflows. We believe event-driven async pipelines are worth the debugging complexity they introduce.
Good architecture is opinionated and evolvable. Opinions that cannot be revised are dogma, not design.
If these beliefs are wrong for your deployment context, the architecture tells you where to change them. The port interfaces are the seams. The event bus is replaceable. The cache strategy is configurable.
