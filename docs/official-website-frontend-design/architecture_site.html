01 Product Vision
A collaborative asset platform is, at its heart, a distributed state synchronization problem with a media delivery layer on top.

Most collaborative asset platforms are feature-accumulation projects — they grow by bolting on capabilities until the architecture collapses under its own surface area. This platform is designed differently: as a small number of deeply considered primitives that compose cleanly.

Teams upload, annotate, version, and curate visual assets together — across time zones, network conditions, and organizational boundaries. The system must make conflict-free concurrency feel natural, AI-assisted enrichment feel invisible, and multi-tenant isolation feel effortless.

02 Design Philosophy
P1
Local-First, Server-Confirmed
Collaboration state is owned locally and synchronized remotely. The server acts as a persistence and relay layer, not an authority on what the document looks like. Latency is physics — round trips make collaboration feel like working through a proxy.
P2
Explicit Over Implicit
Every system boundary is explicit. Storage providers implement a named interface. Cache layers have defined eviction contracts. AI pipeline stages emit typed events. Nothing is ambient.
P3
Isolation Without Ceremony
Tenant context is propagated through a scoped execution context, not threaded manually through every function signature. The isolation model is enforced at the persistence layer boundary, not sprinkled across business logic.
P4
AI as Infrastructure, Not Feature
AI tagging, captioning, and deduplication are pipeline stages that run unconditionally on every ingested asset — not user-facing features with toggle switches. The user sees accurate tags and deduplicated results without invoking anything explicitly.
P5
Stateless Edge, Stateful Core
The HTTP tier is stateless. JWTs carry identity. Redis carries session-adjacent state. PostgreSQL carries durable state. Nothing that matters lives in the application process. Horizontal scaling is a deployment decision, not an architectural one.
03 Core Architecture Pillars
Collaboration
Yjs CRDT + STOMP/WebSocket
Convergence without coordination. Conflicts are mathematically impossible, not just unlikely.
Media Delivery
Caffeine → Redis → CDN
Sub-millisecond hot reads via L1 heap cache. Cost-efficient cold reads via L2 Redis.
AI Enrichment
Async typed event pipeline
Non-blocking ingest path. Every pipeline stage is retryable and independently observable.
Storage
ObjectStoragePort interface
COS/local adapters. Storage backend is a configuration decision, not an architectural one.
Tenancy
Row-level + schema-per-tier
Hibernate filter enforces tenantId on every session. Schema isolation for regulated tenants.
Observability
Logs + Micrometer + OTEL
Structured logs with correlation IDs. Business metrics per tenant. Distributed trace causality.
05 Architecture Overview
06 Realtime Collaboration Model
CRDTs do not require coordination — any two clients can merge states independently and arrive at the same result. OT requires a central server to maintain canonical operation order. That server becomes a coordination bottleneck.

For asset metadata editing — tags, descriptions, annotations, category assignments — the Yjs document model maps cleanly. Concurrent edits to different fields never conflict. Concurrent edits to the same field use last-write-wins semantics at the character level.

Awareness protocol: Cursor positions and presence indicators are ephemeral — transmitted as Yjs Awareness updates but stored in Redis with a 30-second TTL, never persisted to the document state.

07 AI Pipeline

Every asset passing through the upload endpoint unconditionally triggers the AI pipeline. There is no opt-in. Tags, deduplication signals, and content policy decisions should be properties of every asset in the system, not optional metadata for well-behaved clients.

①
HTTP POST /assets — synchronous path
Validate JWT, resolve tenant context, stream binary to ObjectStorage. Returns AssetId the moment storage durability is confirmed. Does not wait for AI.
latency target: P95 < 800ms
②
AssetIngested event — handoff to pipeline
Published immediately after storage write. All pipeline stages subscribe to this event. Workers execute independently; stages with no data dependencies run in parallel.
event: { assetId, storageKey, tenantId, contentType }
③
Pipeline stages — async, retryable
Perceptual hashing, AI tagging (Qwen-VL), content moderation, thumbnail generation. Each stage emits its own completion event for downstream consumers (search indexer, notification center, audit log).
idempotent by design — safe to re-run on failure
①
Download thumbnail from ObjectStorage
The tagging worker fetches the 400w WebP thumbnail, not the original. This reduces API payload size and Qwen-VL inference cost without meaningfully degrading tag quality for content categorization tasks.
②
ChatClient.call(image + structured prompt)
Spring AI's ChatClient abstraction targets Qwen-VL. The prompt requests a JSON array of tags with confidence scores. Response is parsed as typed output.
response: [{"tag": "outdoor", "confidence": 0.94}, ...]
③
Taxonomy normalization per tenant
Raw tags are normalized against the tenant's configured taxonomy before writing. Enables per-tenant tag vocabularies — a legal team sees "contract", "clause"; a design team sees "brand", "illustration" — without changing the pipeline code.
④
Batch INSERT to asset_tags
All normalized tags written in a single batched statement. Emits AssetTagged event consumed by the search indexer and notification center.
①
Normalize to 32×32 grayscale luminance
The original image is downsampled to a 32×32 thumbnail operating on the luminance channel. This removes color, resolution, and minor encoding differences from the comparison surface.
②
Compute 64-bit pHash
DCT-based perceptual hash produces a 64-bit fingerprint. Stored alongside the asset record.
column: assets.phash BIGINT NOT NULL
③
Nearest-neighbor query (Hamming distance ≤ 4)
Catches re-encoded JPEGs, resized versions, and minor crops — not just byte-identical files. Threshold of 4 bits empirically avoids false positives for similar-but-distinct images while catching true perceptual duplicates.
result: duplicate_of FK | null
09 Consistency Model

The system does not chase global strong consistency. It chooses the consistency level appropriate to each operation's failure semantics, and is explicit about those choices in the data model and API contracts.

11 Cache Strategy

Write strategy: write-around. Cache is populated on read, not on write. Newly uploaded assets do not pre-warm the cache — the first access triggers population. This avoids cache churn on batch uploads.

13 Observability
Observability is not logging. Logs record events. Metrics record rates. Traces record causality. All three are required to answer: why did P95 latency spike for tenant X at 14:32?

Every inbound request receives a X-Request-ID header. Every async event carries a correlationId derived from the originating request. Every log line includes both. This makes it possible to trace a single upload across the synchronous API response and every async pipeline stage that follows.

14 Future Roadmap
Near Term
Asset versioning with diff visualization
Content-addressed binary deltas between versions. Perceptual diffs for image assets.
Webhook delivery — at-least-once
Persistent outbox table, exponential backoff retry, dead-letter queue.
Full-text search
Elasticsearch or PostgreSQL tsvector for tag + description search within tenant scope.
Medium Term
Pluggable event bus
Formalize DomainEventPublisher with a Kafka adapter. Zero business logic changes required.
AI model routing
Route tagging to Qwen-VL, GPT-4o-Vision, or Gemini by tenant tier or content type.
Edge presigned URLs
CDN-signed tokens enabling edge caching of authenticated asset delivery.
Long Term
Federated collaboration
Cross-tenant asset sharing via capability-based link model with explicit grant scoping and audit trail.
WASM-accelerated CRDT
Compile Yjs merge functions to WASM, reducing GC pressure in the relay tier.
Differential sync
Vector-clock-based differential sync on reconnect, reducing payload for large documents.
15 Non-Goals
✕Video/audio processing. The pipeline is designed for still images. Video ingest, transcoding, and streaming require a separate media processing tier.
✕Public CDN hosting. Presigned URLs provide time-bounded authenticated access. Anonymous public delivery is not a design target.
✕Branching or merge workflows. Asset versioning is linear. Git-style branching of metadata trees is not modeled.
✕Cross-region replication. The storage abstraction supports plugging in a multi-region backend, but the application layer does not implement region-aware routing.
✕Real-time annotation drawing. Yjs-backed text metadata editing is in scope. Pixel-level collaborative drawing (Figma-style canvas) is not.
16 Trade-offs
Decision Upside Downside + Mitigation
In-process event bus over Kafka Zero infra dependency. No consumer lag. No broker on upload path. Events lost on process crash. Mitigated by idempotent pipeline stages and admin requeue endpoint. DomainEventPublisher interface is the Kafka escape hatch.
JWT over server sessions Stateless pods. No sticky sessions. Tenant context in token, no per-request DB lookup. Token revocation requires short expiry or Redis revocation set. Current: 24h expiry + Redis revocation set checked only on sensitive operations.
Yjs over ShareDB / Phoenix Battle-tested CRDT, strong TypeScript support, offline resilience, rich type ecosystem. Binary opaque state complicates ops inspection. Server-side CRDT requires GraalVM Polyglot to run Yjs merge function. Non-trivial operational overhead.
Row-level tenant isolation Operationally simple. One pool, one migration path, one schema version. Misconfigured Hibernate filter could expose cross-tenant data. Mitigated by integration tests asserting isolation on every tenant-scoped query. Schema-per-tenant available for enterprise tier.
