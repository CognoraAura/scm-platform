Why We Build This Way

Opinion is architecture. Behind every technical decision lies a value judgment.

IDomain-Driven Design
When the domain model is correct, code reads like business language. When it's wrong, every line of code is a translation.

We don't worship DDD as methodology — we use it as a communication tool. In the supply chain domain, terminology is territory — "order" means something very different in procurement versus sales contexts. We use Bounded Contexts to ensure each service owns its semantic space, rather than sharing a bloated, anemic model.

Aggregate root design follows the "consistency boundary" principle: one transaction modifies one aggregate. The order aggregate includes order lines and payment records, but not logistics tracking — that belongs to a different context. This boundary constraint forces eventual consistency between services rather than strong transactions — a technical decision that is also a business decision.

The Repository layer abstracts persistence details, letting domain models focus on business rules. We oppose exposing JPA annotations or ORM-specific logic in the domain layer — domain objects shouldn't know whether they're persisted in PostgreSQL or Redis.

IICQRS

Read-write separation isn't a "performance optimization" — it's a modeling choice. In the scm-system module, the complexity of cross-database queries forced us to confront the limitations of a unified model. CQRS lets us build optimal models for reads and writes separately: the write model follows domain rules, the read model follows query efficiency.

The Command side uses JPA + @Master to write to the primary database, while the Query side uses MyBatis-Plus + @Slave to read from replicas or Redis cache. Both are synchronized through events — when the write model changes, a DomainEvent is published, and Eventual Consistency automatically updates the read model.

Trade‑off
rejected Unified model (CRUD Repository handling both queries and commands) — cross-database joins cause unpredictable performance, query logic pollutes the domain model.
chosen CQRS separation — each use case's queries can be independently optimized while the domain model stays clean.
IIIEvent-Driven
An event is not a message — it's an irreversible fact. Once persisted, it becomes part of the audit log.

In distributed systems, synchronous call chains are a source of fragility. We use Spring Cloud Function with a message broker for event-driven orchestration: when an order is created, the system publishes OrderCreatedEvent, and inventory, logistics, and notification services respond independently. No central orchestrator, no synchronous blocking.

We follow event sourcing principles — not by storing all state as event streams, but by persisting critical business events to dedicated event tables as a foundation for auditing and diagnostics. Kafka serves as the event backbone, guaranteeing at-least-once delivery, with consumers handling duplicate messages through idempotency.

Trade‑off
rejected Synchronous pipeline (inter-service orchestration via HTTP) — lacks resilience, a single downstream failure blocks the entire chain.
cost Events can be lost (message broker failure, consumer crash, network partition) — mitigated through dead letter queues, scheduled compensation, and idempotency tables.
IVCache as Performance Architecture

Caching is not a post-hoc optimization — it's a first-class architectural concern. We design explicit caching strategies for every query path, rather than reactively adding caches after performance bottlenecks emerge.

01
L1 Caffeine Local Cache
Each service instance maintains a local cache with a 5-minute TTL, reducing reads on Redis. Suitable for dictionary data, configuration items, and domain objects with low change frequency.
02
L2 Redis Distributed Cache
Cross-service shared cache data with a 30-minute TTL. Uses Redis Pub/Sub to actively invalidate related caches on data changes, rather than relying on passive TTL expiration.
03
Strong Invalidation Strategy
Invalidate cache immediately after write operations complete, rather than updating the cache. The next read request naturally refills with fresh data, avoiding cache-to-database inconsistency windows.
VStorage Abstraction
Storage selection is one of the most regret-inducing and costly architectural decisions. An abstraction layer preserves your right to change it.

We don't use RedisTemplate or JPA EntityManager directly in business code. Instead, every data access goes through Repository interfaces or custom abstraction layers. This not only simplifies testing — more importantly, when the day comes to migrate from PostgreSQL to CockroachDB, or upgrade from Redis Standalone to Redis Cluster, the scope of change is contained within a single layer.

Trade‑off
rejected Direct SDK integration (business code calling RedisTemplate, MongoTemplate, etc.) — code deeply couples with specific storage, changing storage means rewriting the service.
cost Each storage abstraction layer adds roughly two hours of development time — but over the project lifecycle, the flexibility gained far outweighs the cost.
VIAI as Infrastructure
AI is not a product feature — it's infrastructure capability. We don't name requirements as "AI features" but instead weave intelligence into the seams of existing business processes.

We don't aim to replace human decision-makers with AI. Models predict, people decide. The AI module handles: demand forecasting, inventory health scoring, supplier risk assessment, and order anomaly detection. These capabilities appear embedded within business interfaces, not in a separate AI console.

01
Non-Blocking Integration
AI inference results are output as JSON structures, consistent with business data formats. The system runs fully even without AI features — AI is additive, not a dependency.
02
Async Inference
All AI inference goes through the message queue for async execution. User operations are unaffected by model inference time — inventory predictions run in the background and results are pushed to the frontend via WebSocket.
03
Classification Normalization
Product classifications from different sources (SPU categories, customs codes, warehouse location types) are mapped to a unified classification tree via AI, eliminating N×N mapping complexity.
VIIMulti-Tenant Isolation
Isolation level is a security decision, not an architectural one. Choose the wrong isolation model and you either leak data or burn out maintaining it.

We adopt a hybrid isolation strategy: logical isolation (shared database filtered by tenant) for most scenarios, and physical isolation (dedicated database instances) for high-security requirements. TenantContextHolder resolves the tenant identifier at the request entry point, threading through the entire call chain to ensure data boundaries remain intact.

01
Hibernate Filters
The ORM layer automatically appends tenant_id filter conditions — developers never manually concatenate SQL. All tables include a tenant_id column with covering indexes.
02
Redis Key Prefixes
Cache keys use tenant_id as a prefix to prevent cross-tenant cache overlap. Hot data like flash-sale inventory is sharded per tenant.
03
Schema Isolation
For finance-grade tenants, use dedicated database schemas rather than row-level filtering. Connection pools route dynamically per tenant, with TenantContextHolder driving @DS annotation-based data source switching.
VIIIScaling Philosophy
Don't predict bottlenecks. Monitor them, then scale. Premature scaling and never scaling are equally dangerous.

Our scaling strategy follows the "vertically then horizontally" principle: before clear horizontal scaling needs emerge, optimize through vertical scaling and code improvements first. When horizontal scaling becomes necessary, each service's stateless design ensures simplicity.

easy
HTTP Layer
Stateless services scale horizontally through load balancing. The Gateway layer auto-routes, Nacos registry detects node changes in real time. Add instances, increase throughput.
hard
WebSocket Layer
Long-lived connections require session affinity. Use Redis Pub/Sub to broadcast cross-instance messages, or configure Sticky Sessions at the Gateway layer.
easy
Worker Layer
Message consumer tasks are stateless, automatically distributed through Kafka partition rebalancing. Increase consumer group instances to boost throughput.
defer
Database
Read-write separation + sharding (ShardingSphere) is already provisioned at the architecture level, but only enabled when a single database reaches 5000 QPS or 2TB.
IXOperations Philosophy

An architectural design that can't run is useless, no matter how elegant. We treat operability as an architectural requirement from day one — meaning every service knows how it will be deployed, monitored, and debugged before a single line of code is committed. Infrastructure as Code (IaC) and immutable infrastructure are our default operations model.

01
Observability First
Every service must expose health checks, metric endpoints (Prometheus), distributed tracing (SkyWalking), and structured logging. No instrumentation, no deployment.
02
Systematic Idempotency
All critical write operations (order creation, stock deduction, payment callbacks) use request IDs for idempotency. Redis stores a 24-hour idempotency table, safely degrading duplicate requests.
03
Graceful Degradation
Sentinel circuit breaking + service degradation is a default configuration, not an optional feature. Every Feign call has a fallback, every cache miss has protection. The system degrades rather than avalanches.
XNon-Goals

Knowing what not to do is harder than knowing what to do. These are things we deliberately avoid — constraints that sharpen our architectural focus.

✕
No pursuit of 100% availability — Our SLO target is 99.9% (~43 minutes downtime per month). This means we don't need multi-region active-active deployment, unitized architecture, or cross-AZ real-time sync. This restraint lets us focus resources on what matters.
✕
No pursuit of "technical novelty" — We prefer mature, battle-tested technologies. Kubernetes is our infrastructure foundation, but we don't adopt Service Mesh; we use PostgreSQL over NewSQL databases; we choose classic message queues over event streaming platforms. Innovation is confined to the business logic layer.
✕
Not a general-purpose platform — SCM Platform is designed for mid-to-large enterprise supply chains, not a general-purpose PaaS. We assume users have dedicated operations teams, clear domain expertise, and customization needs. We don't aim to be an "out-of-the-box" SaaS product.
✕
No promise of "zero configuration" — Good architecture requires understanding. We don't try to hide complexity behind excessive abstractions — every service's startup order, configuration meaning, and database initialization scripts need to be understood. Documentation doesn't replace knowledge, but it lowers the barrier to acquiring it.
