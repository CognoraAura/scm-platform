design philosophy · v1.0

我们为什么这样构建

观点即架构。每一个技术决策背后都是一个价值判断。

I领域驱动设计
当领域模型正确时，代码读起来像业务语言。当领域模型错误时，每一行代码都是一次翻译。

我们不把 DDD 当作方法论来崇拜，而把它当作沟通工具。在供应链领域，术语本身就是疆域——"订单"在采购语境和销售语境中含义截然不同。我们通过限界上下文（Bounded Context）确保每个服务拥有自己的语义空间，而不是共享一个臃肿的贫血模型。

聚合根的设计遵循"一致性边界"原则：一个事务只修改一个聚合。订单聚合包含订单行和支付记录，但不包含物流轨迹——后者属于另一个上下文的职责。这种边界约束迫使我们在服务间使用最终一致性，而非强事务，这是技术决策，更是业务决策。

Repository 层隐藏持久化细节，让领域模型专注于业务规则。我们反对在领域层暴露 JPA 注解或 ORM 特化逻辑——领域对象不该知道它被存在 PostgreSQL 还是 Redis 里。

IICQRS

读写分离不是一个"性能优化"，而是一种建模选择。在 scm-system 模块中，跨数据库查询的复杂性迫使我们正视统一模型的局限性。CQRS 让我们为读操作和写操作分别建立最优模型：写模型服从领域规则，读模型服从查询效率。

Command 端使用 JPA + @Master 写主库，Query 端使用 MyBatis-Plus + @Slave 读从库或 Redis 缓存。两者通过事件同步——当写模型变更时，发布 DomainEvent，Eventual Consistency 自动更新读模型。

Trade‑off
rejected 统一模型（CRUD Repository 同时承担查询和命令）—— 跨数据库 join 时性能不可控，查询逻辑污染领域模型。
chosen CQRS 分离 —— 每个用例的查询可以独立优化，领域模型保持纯净。
III事件驱动
事件不是消息，是不可撤销的事实。一旦入库，它就是审计日志的一部分。

分布式系统中，同步调用链是脆弱性的源头。我们使用 Spring Cloud Function + 消息中间件实现事件驱动编排：当订单创建后，系统发布 OrderCreatedEvent，库存服务、物流服务、通知服务分别独立响应。没有中心编排器，没有同步阻塞。

我们遵循"事件溯源"的思想——不是必须把所有状态存成事件流，而是把关键业务事件持久化到独立的事件表，作为审计和诊断的依据。Kafka 作为事件主干，保证至少一次投递，消费者通过幂等性处理重复消息。

Trade‑off
rejected 同步管道（服务间通过 HTTP 同步调用编排）—— 缺乏弹性，一个下游故障阻塞整个链路。
cost 事件可能丢失（消息中间件故障、消费者崩溃、网络分区）—— 通过死信队列 + 定时补偿 + 幂等表兜底。
IV缓存即性能架构

缓存不是事后追加的优化，而是架构的一等公民。我们为每一个查询路径设计了明确的缓存策略，而非在性能瓶颈出现后才被动加缓存。

01
L1 Caffeine 本地缓存
每个服务实例维护 5 分钟 TTL 的本地缓存，减少对 Redis 的读取压力。适合字典数据、配置项、低频变更的领域对象。
02
L2 Redis 分布式缓存
跨服务共享的缓存数据，30 分钟 TTL。使用 Redis 发布/订阅机制在数据变更时主动失效相关缓存，而非依赖 TTL 被动过期。
03
强失效策略
写操作执行完毕后立即失效缓存，而非更新缓存。下一次读请求自然回填最新数据，避免缓存与数据库的不一致窗口。
V存储抽象
存储选型是架构中最后悔成本最高的决策之一。抽象层让你保留换掉它的权利。

我们不直接在业务代码中使用 RedisTemplate 或 JPA EntityManager。相反，每个数据访问都通过 Repository 接口或自定义抽象层进行。这不仅让测试更简单，更重要的是——当某天需要从 PostgreSQL 迁移到 CockroachDB，或者从 Redis Standalone 升级到 Redis Cluster 时，改动范围被限制在一层之内。

Trade‑off
rejected 直接 SDK 集成（业务代码调用 RedisTemplate、MongoTemplate 等）—— 代码与具体存储深度耦合，换存储等于重写服务。
cost 每加一个存储抽象层约额外两小时开发时间 —— 但在项目生命周期内，这种抽象带来的灵活性回报远超成本。
VIAI 作为基础设施
AI 不是产品功能，是基础设施能力。我们不以「AI 功能」来命名需求，而是把智能能力编织到现有业务流程的缝隙里。

我们不追求用 AI 替换人类决策者。模型做预测，人做决策。AI 模块负责：需求预测、库存健康评分、供应商风险评级、订单异常检测。这些能力以嵌入式的形式出现在业务界面中，而非单独的 AI 控制台。

01
无条件集成
AI 推理结果以 JSON 结构输出，与业务数据格式一致。即使用户不使用 AI 功能，业务系统完全可运行——AI 是增量，不是依赖。
02
异步推理
所有 AI 推理走消息队列异步执行。用户操作不受模型推理时间影响——库存预测在后台完成，结果通过 WebSocket 推送到前端。
03
分类归一化
不同来源的商品分类（SPU 分类、海关编码、仓库货位分类）通过 AI 映射到统一分类树，消除 N×N 的映射复杂度。
VII多租户隔离
隔离级别是一个安全决策，不是架构决策。选错了隔离模型，要么漏数据，要么维护到崩溃。

我们采用混合隔离策略：大多数场景使用逻辑隔离（共享数据库按租户过滤），高安全需求场景使用物理隔离（独立数据库实例）。TenantContextHolder 在请求入口处解析租户标识，贯穿整个调用链路，确保数据边界不被突破。

01
Hibernate 过滤器
ORM 层自动追加 tenant_id 过滤条件，开发者无需手动拼接 SQL。所有表均包含 tenant_id 列，索引覆盖过滤条件。
02
Redis 键前缀
缓存键使用 tenant_id 作为前缀，确保不同租户的缓存数据不会互相覆盖。秒杀库存等热数据按租户分片存储。
03
Schema 隔离
对于金融级租户，使用独立数据库 Schema 而非行级过滤。连接池按租户动态路由，TenantContextHolder 驱动 @DS 注解切换数据源。
VIII扩展哲学
不要预测瓶颈。监控它，然后扩展它。过早扩展和从不扩展一样危险。

我们的扩展策略遵循"先垂直再水平"的原则：在出现明确水平扩展需求之前，先通过垂直扩容和代码优化释放性能。当必须水平扩展时，每个服务的无状态设计确保了扩展的简洁性。

easy
HTTP 层
无状态服务通过负载均衡水平扩展。Gateway 层自动路由，Nacos 注册中心实时感知节点上下线。加实例，提吞吐。
hard
WebSocket 层
长连接服务需要会话亲和性。通过 Redis Pub/Sub 广播跨实例消息，或在 Gateway 层做 Sticky Session。
easy
Worker 层
消息消费者任务无状态，通过 Kafka 分区重平衡自动分配。增加消费者组实例数即可提升吞吐。
defer
数据库
读写分离 + 分库分表（ShardingSphere）已在架构层预留，但只在单库达到 5000 QPS 或 2TB 时启用。
IX运维哲学

一个跑不起来的架构设计再优雅也毫无意义。我们从第一天就把可运维性作为架构需求——这意味每个服务在提交代码之前就已经想好了如何部署、如何监控、如何排障。基础设施即代码（IaC）和不可变基础设施是我们默认的运维模型。

01
可观测性优先
每个服务必须暴露健康检查、指标端点（Prometheus）、分布式追踪（SkyWalking）和结构化日志。没有埋点的代码不允许上线。
02
幂等性系统化
所有关键写操作（订单创建、库存扣减、支付回调）通过请求 ID 实现幂等性。Redis 存储 24 小时幂等表，重复请求安全降级。
03
优雅降级
Sentinel 熔断 + 服务降级是默认配置，不是可选功能。每个 Feign 调用都有 fallback，每个缓存穿透都有保护。系统宁可降级也不可雪崩。
X非目标

明确不做什么比明确做什么更难。以下是我们刻意不做的事情——这些限制让我们的架构选择更加专注。

✕
不追求 100% 可用性 — 我们的 SLO 目标是 99.9%（每月约 43 分钟停机）。这决定了我们不需要多活部署、不需要单元化架构、不需要跨 AZ 实时同步。在可用性上的克制让我们能集中资源做正确的事。
✕
不追求"技术先进性" — 我们倾向成熟、经过验证的技术栈。Kubernetes 是基础设施基座，但不引入 Service Mesh；使用 PostgreSQL 而非 NewSQL 数据库；选择经典的消息队列而非事件流平台。创新的边界只停留在业务逻辑层。
✕
不做通用平台 — SCM Platform 是为中大型供应链企业设计的，不是通用型 PaaS。我们假设用户有专门的运维团队、有明确的业务领域知识、有定制化的需求。不做"开箱即用"的 SaaS 产品。
✕
不承诺"零配置" — 好的架构需要理解。我们不试图用过度抽象隐藏复杂度——每个服务的启动顺序、配置项含义、数据库初始化脚本都需要被理解。文档不替代知识，但降低获取知识的门槛。
